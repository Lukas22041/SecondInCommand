## 1. Project Overview

This is the static frontend / wiki site for the **Second-in-Command** Starsector mod.  
It is a standalone React app (no build step, CDN Babel) that reads `api/api.json` — a JSON
file exported by the mod itself — and renders every aptitude and skill with full tooltip data.

**Source mod:** [`SecondInCommand`](https://github.com/Lukas22041/SecondInCommand)  
**Data origin:** The mod's `SCBuildScript` Kotlin class generates `api/api.json` at runtime inside
the game, then this frontend consumes it statically.

## 2. API Schema (`api/api.json`)

The file is generated by `SCBuildScript.kt` and serialised by `SCBuildDataModels.kt`.
All image paths inside the JSON are relative to the `api/` folder (e.g. `"assets/foo.png"`).
The frontend's `resolveApiPath()` utility in `js/utils.js` prepends `api/` to these paths
so they resolve correctly from the page root.

---

### Top level

```json
{
  "version":      "string  — mod version string (e.g. \"1.4.2\")",
  "exportDate":   "string  — ISO-8601 timestamp of when the export ran",
  "tooltipWidth": "number  — pixel width used when rendering tooltips in-game (always 700)",
  "aptitudes":    "AptitudeData[]"
}
```

---

### `AptitudeData`

One entry per aptitude registered with the mod.

```json
{
  "id":            "string  — unique aptitude ID (e.g. \"sc_logistics\")",
  "name":          "string  — display name",
  "description":   "string  — plain text from addCodexDescription(); may be empty",
  "color":         "ColorData — theme colour used for borders, glows, badges",
  "categories":    "CategoryData[]",
  "tags":          "string[] — see tag list below",
  "order":         "number  — sort order across all aptitudes (lower = first)",
  "modName":       "string  — human-readable mod name (e.g. \"Second-in-Command\")",
  "originSkillId": "string  — ID of the always-unlocked first skill",
  "sections":      "SectionData[]"
}
```

**Known tags** (frontend filters on these):

| Tag | Effect |
|-----|--------|
| `hide_in_codex` | Aptitude is hidden in the in-game codex and on this site |
| `dont_include_in_wiki` | Aptitude is excluded from this site's render |

The first `SectionData` entry always contains exactly one skill (the origin skill) with
`canChooseMultiple: true` and `requiredPreviousSkills: 0`.

---

### `CategoryData`

Aptitudes can belong to a mutual-exclusion category (e.g. "Combat"). Only one aptitude
per category may be active at a time.

```json
{
  "id":    "string  — unique category ID",
  "name":  "string  — display name",
  "color": "ColorData"
}
```

When more than one aptitude in the rendered list shares a category ID, the frontend
shows a category notice beneath that aptitude row.

---

### `SectionData`

A horizontal row of skills within an aptitude. Aptitudes have one or more sections
separated by chevron dividers in the UI.

```json
{
  "canChooseMultiple":      "boolean — false = pick-one (shown with underline bracket)",
  "requiredPreviousSkills": "number  — how many skills from the prior section must be unlocked",
  "skills":                 "SkillData[]"
}
```

---

### `SkillData`

```json
{
  "id":              "string  — unique skill ID",
  "name":            "string  — display name",
  "iconPath":        "string  — original in-game sprite path (e.g. \"graphics/icons/...\")",
  "iconExportPath":  "string  — relative path to the exported PNG (e.g. \"assets/foo.png\")",
  "assetFileName":   "string  — bare filename of the exported PNG (e.g. \"foo.png\")",
  "affectsString":   "string  — short label such as \"Fleet\" or \"Officers\"",
  "order":           "number  — sort order within its section",
  "modName":         "string  — which mod registered this skill",
  "isOriginSkill":   "boolean — true for the first skill in the aptitude (always unlocked)",
  "tooltipElements": "TooltipElement[]"
}
```

`iconExportPath` is what the frontend uses as the `<img src>` (after `resolveApiPath()`).

---

### `TooltipElement` types

Tooltip elements are rendered in **y-descending order** — Starsector's coordinate system
has y increasing upward, so higher y = closer to the top of the tooltip.

All element types carry a `position` field of type `PositionData`.

#### `title`

The bold heading at the top of a tooltip.

```json
{
  "type":             "\"title\"",
  "text":             "string",
  "color":            "ColorData",
  "highlightRanges":  "HighlightRange[]",
  "font":             "string | null — font method name if non-default",
  "padding":          "number — top padding in pixels",
  "position":         "PositionData"
}
```

#### `label`

A body-text paragraph. May contain inline colour highlights.

```json
{
  "type":             "\"label\"",
  "text":             "string — may contain embedded \\n newlines",
  "color":            "ColorData — base text colour",
  "highlightRanges":  "HighlightRange[]",
  "font":             "string | null",
  "padding":          "number",
  "position":         "PositionData"
}
```

#### `spacer`

Vertical whitespace gap.

```json
{
  "type":     "\"spacer\"",
  "height":   "number — gap height in pixels",
  "position": "PositionData"
}
```

#### `imageWithText`

A skill ability image (e.g. a ship system icon) displayed alongside child text elements.

```json
{
  "type":          "\"imageWithText\"",
  "spriteName":    "string — original in-game sprite path",
  "exportPath":    "string — relative path (e.g. \"assets/foo.png\"); empty string if no image",
  "assetFileName": "string",
  "imageHeight":   "number — render height in pixels",
  "padding":       "number",
  "children":      "TooltipElement[] — title / label / spacer elements rendered beside the image",
  "position":      "PositionData"
}
```

#### `sectionHeading`

A styled section-divider bar (used inside some complex tooltips).
Currently parsed but **not rendered** by the frontend.

```json
{
  "type":      "\"sectionHeading\"",
  "text":      "string",
  "textColor": "ColorData",
  "bgColor":   "ColorData",
  "alignment": "string",
  "padding":   "number",
  "position":  "PositionData"
}
```

#### `image`

A standalone image (no flanking text). Currently parsed but **not rendered** by the frontend.

```json
{
  "type":          "\"image\"",
  "spriteName":    "string",
  "exportPath":    "string",
  "assetFileName": "string",
  "width":         "number",
  "height":        "number",
  "padding":       "number",
  "position":      "PositionData"
}
```

---

### Shared primitives

#### `ColorData`

All values are integers in the range 0–255.

```json
{ "r": 255, "g": 128, "b": 0, "a": 255 }
```

#### `PositionData`

Pixel coordinates from the in-game tooltip panel.
**y is bottom-up** (Starsector game convention): a higher `y` value means the element
is positioned higher on screen. Sort descending on `y` to get top-to-bottom render order.

```json
{ "x": 0.0, "y": 540.0, "width": 700.0, "height": 24.0 }
```

#### `HighlightRange`

Marks a substring within a `label` or `title` element that should be rendered in a
different colour.

```json
{
  "startIndex": 12,
  "endIndex":   20,
  "text":       "the highlighted substring",
  "color":      "ColorData"
}
```

Indices are character positions within the parent element's `text` string.
Multi-line labels (those containing `\n`) require per-line index adjustment before use —
see `HighlightedText` in `js/tooltip.js`.

## 3. Frontend Architecture

### Technology

- **React 18** + **ReactDOM** loaded from unpkg CDN
- **Babel Standalone** — transpiles JSX in-browser, no build step
- Scripts use `<script type="text/babel">` and share a single global scope
- Load order in `index.html` replaces ES module imports

### Data flow

```
fetch('./api/api.json')
  └─ App (js/app.js)
       ├─ useMemo: filter hidden aptitudes, split SIC vs third-party, build sharedCategoryIds
       └─ render tree (see component tree below)
```

`resolveApiPath(p)` in `js/utils.js` is called whenever an image `src` is derived from the
JSON. It prepends `api/` to any path starting with `assets/`, translating the JSON's
relative paths to paths that resolve from the page root.

### Component tree

```
App
├── SideNav                         (js/nav.js)
├── page-banner <img>
├── AboutSection                    (js/sections.js)
│   └── GalleryCarousel             (js/gallery.js)
│       └── lightbox portal → document.body
├── LinksSection                    (js/sections.js)
├── IncompatibilitiesSection        (js/sections.js)
└── section#aptitudes
    ├── SectionHeading              (js/ui.js)
    ├── [SIC aptitudes]
    │   └── AptitudeRow × N         (js/aptitude.js)
    │       ├── AnchorCopy          (js/ui.js)
    │       ├── SectionSeparator × N (js/skill.js)
    │       └── SkillSection × N    (js/skill.js)
    │           └── SkillIcon × N   (js/skill.js)
    │               └── TooltipOverlay portal → document.body  (js/tooltip.js)
    │                   └── TooltipRenderer
    │                       ├── title / label (HighlightedText)
    │                       ├── spacer
    │                       └── TooltipImageWithText
    └── [Third-party aptitudes] (same structure, only if present)
```

### File reference

| File | Exports / defines | Notes |
|------|------------------|-------|
| `js/utils.js` | `rgbaToCSS`, `resolveApiPath`, `_ptrX/_ptrY`, React hook destructure | Must load first; defines all globals used by every other file |
| `js/tooltip.js` | `HighlightedText`, `TooltipLabel`, `TooltipImageWithText`, `TooltipRenderer`, `TooltipOverlay` | `TooltipOverlay` uses a React portal to `document.body` so it escapes any overflow:hidden containers |
| `js/ui.js` | `AnchorCopy`, `SectionHeading` | `AnchorCopy` copies a `#anchor` URL to clipboard; used both on section headings and aptitude titles |
| `js/skill.js` | `SkillIcon`, `SectionSeparator`, `SkillSection` | `SkillIcon` suppresses tooltips during scroll and re-checks pointer position 150 ms after scroll stops |
| `js/aptitude.js` | `AptitudeRow` | Shows category notice when `sharedCategoryIds` contains the aptitude's category |
| `js/gallery.js` | `GALLERY_IMAGES`, `GalleryCarousel` | 3-up carousel with a full-screen lightbox portal; images live in `appAssets/` |
| `js/sections.js` | `AboutSection`, `LinksSection`, `IncompatibilitiesSection` | Static content sections; `AboutSection` embeds `GalleryCarousel` |
| `js/nav.js` | `SIDE_NAV_SECTIONS`, `SideNav` | Sticky left nav; tracks the active section via a scroll listener with a 90 px offset |
| `js/app.js` | `App` + `ReactDOM.createRoot` mount | Fetches data, derives aptitude lists, owns all top-level state |

## 4. Build Pipeline (Mod → API)

### Overview

The build pipeline lives entirely inside the mod's Java/Kotlin source under
`src/second_in_command/buildscript/`. It runs **inside the Starsector game process**
on the first frame after loading a save, then writes its output to disk and exits.

```
Game load
  └─ SCModPlugin.onGameLoad()
       └─ (if SC_DEV_BUILD env var is set)
            └─ SCBuildScript  [EveryFrameScript, runs once]
                 ├─ SCBuildScript.buildExportData()
                 │    ├─ Iterates SCSpecStore.getAptitudeSpecs()
                 │    ├─ For each skill: renders tooltip via SCSkillTooltipCreator
                 │    │    └─ through RecordingTooltipMaker (proxy)
                 │    └─ TooltipElementParser.parse() → TooltipElement list
                 ├─ SCBuildExporter.export()
                 │    ├─ Writes  <mod>/build_export/api.json
                 │    └─ Copies  <mod>/build_export/assets/*.png
                 └─ done = true  (script removes itself)
```

### Key classes

| Class | File | Role |
|-------|------|------|
| `SCBuildScript` | `buildscript/SCBuildScript.kt` | Entry point; orchestrates the whole export |
| `SCBuildDataModels` | `buildscript/SCBuildDataModels.kt` | Kotlin data classes that mirror the JSON schema; each has a `toJSON()` method |
| `SCBuildExporter` | `buildscript/SCBuildExporter.kt` | Writes `api.json` and copies sprite PNGs using reflection to bypass Starsector's class-loader file restrictions |
| `RecordingTooltipMaker` | `buildscript/TooltipRecordingProxy.kt` | Kotlin `by`-delegation proxy around `TooltipMakerAPI`; records every method call so that highlight colours, padding, and sprite paths can be extracted after rendering |
| `TooltipElementParser` | `buildscript/TooltipElementParser.kt` | Converts the list of `RecordedCall` objects into `TooltipElement` instances; extracts positions from the return values of the real tooltip calls |

### How tooltip data is captured

Starsector's `TooltipMakerAPI` renders UI into a panel but exposes no getters for
highlight ranges, padding values, or sprite names after the fact.
`RecordingTooltipMaker` wraps the real API via Kotlin interface delegation (`by real`).
Every call is forwarded to the real implementation (so the UI actually renders) **and**
is recorded as a `RecordedCall(methodName, args, result)`.

`TooltipElementParser.parse()` then walks those recorded calls:
- `addTitle` / `addPara` → `LabelElement` (title flag set for `addTitle`)
- `addSpacer` → `SpacerElement`
- `addImageWithText` → `ImageWithTextElement` (children parsed from inner recorded calls)
- `addSectionHeading` → `SectionHeadingElement`
- `addImage` / `addImages` → `ImageElement`
- Font-setter calls (`setParaFont`, etc.) update a `currentFont` variable but emit no element
- Positions are read from the call's return value (a `UIComponentAPI`) rather than
  `getChildrenCopy()`, which was unreliable for `addPara` calls

### Triggering a build

1. Set the environment variable `SC_DEV_BUILD` to any value before launching Starsector.
2. Start the game and load any save.
3. On the first frame the build runs and writes output to `<mod>/build_export/`.
4. Check the game log for `"SC Build: Finished in Xms"` to confirm success.

### Copying output to the frontend

After a successful build, copy the two outputs into the frontend's `api/` folder:

```
build_export/api.json   →  frontend/api/api.json
build_export/assets/    →  frontend/api/assets/
```

The `build_export/` directory is not part of the frontend repo.
